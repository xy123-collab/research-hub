# 勘误规则（三类）

平台有三类"勘误"，流转不同。分别按 `api/bugs.py`、`api/datasets.py`、`api/code.py` 实现书写。

## 一、原始数据勘误（最复杂，`api/bugs.py`）

对象：数据集某条记录某变量的取值。**含历史/评分/自动改数**。

流程：
1. 提交：单条 `POST /datasets/{slug}/bugs`，或批量 `POST .../bugs/batch`（Excel/CSV，先下 `bug-template`）。
   无论单条还是批量，都集成为**一条 Bug + 多个 BugItem**（子项含 `唯一ID值/变量名/当前值/建议值/说明`）。
2. 打分：`POST /bugs/{bid}/ai-review` 或 `/bug-items/{iid}/ai-review`（AI 评分）+ 人工评审 `POST /bugs/{bid}/reviews`。
3. **终审**：`POST /bugs/{bid}/finalize` 或 `/bug-items/{iid}/finalize`，`adopt_level ∈ (full|partial|reject)`；
   非 reject → `accepted`，reject → `rejected`。父 Bug 状态按子项聚合。
4. **自审规则（八节 4）**：`self_review = (reporter == 当前管理员)`；若该数据集**还有其他管理员**，本人提交的勘误
   **必须由另一名管理员终审**（否则 403）；只有一名管理员时允许自审并写审计 `self_review`。
5. 应用改数：`POST /datasets/{slug}/apply-corrections` 把"已采纳未应用"子项按 `唯一ID+变量名` 改基准版 .dta →
   生成新原始版本，子项标 `fixed`、记 `fixed_in_version_id`（见 `03`）。

采纳会给报告人按终审分加贡献；评审方向与终审一致者 +k。

## 二、Codebook / 对照表 勘误（简单，`api/datasets.py`）

对象：某版本的 codebook 或对照表文件。**刻意做简单：不记历史版本迭代**。

流程（`FileCorrection` 表）：
1. 成员提交：`POST /datasets/{slug}/file-corrections`（`target ∈ codebook|mapping` + 文字内容）。前端在版本库每个文件旁的「报错」入口。
2. 管理员查看：`GET /datasets/{slug}/file-corrections`（管理员看全部、成员看自己提交的）。进消息中心待办。
3. 采纳/驳回：`POST .../file-corrections/{fid}/decide?approve=` → `accepted` / `rejected`
   （与原始数据勘误一样的"确认采纳"动作，但没有评分、没有自动改文件、没有版本迭代）。

## 三、处理代码勘误（`api/code.py`）

对象：处理代码脚本。两种：
- **评论式勘误**：`POST /code/{cid}/comments`（`is_correction=True` 标记为勘误类评论）。
- **代码 Bug**：`POST /code/{cid}/bugs` → `POST /codebugs/{bid}/reviews` → `/codebugs/{bid}/finalize`。
代码本身走版本迭代（`code_versions`，发布必须写 changelog），作者可 `POST /code/{cid}/grants` 授予他人"可改/可重发布"。

## 四、共同点

三类勘误的"是否采纳"都由**数据集管理员**拍板；都会进消息中心待办（见 `07`）。区别在于：只有**原始数据勘误**有
评分制、自动改数与版本联动；文件勘误与代码评论式勘误是轻量确认。
