# 人工测试指南 · 科研数据协作平台

> 给测试同学：本文告诉你**从哪些方面测、每个模块重点看什么、按什么格式反馈**。
> 平台一句话：让自建数据可信、可迭代、可复用——围绕「数据集」，做版本管理、勘误纠错、
> 只读在线分析、文献地图、权限与协作。线上地址以对接人给你的为准。

---

## 一、先明确"测什么"——四个维度

每条反馈请归到下面某一维度，方便我们分优先级：

1. **功能实现性（能不能用）**：该功能是否按预期完成。点了没反应、报错、数据没保存、算错了，都属于这一类。**最高优先级。**
2. **正确性与一致性（对不对）**：显示的数字/权限/状态和实际是否一致；同一信息在不同页面是否矛盾；刷新后是否还在。
3. **使用舒适度（顺不顺）**：流程是否绕、提示是否清楚、加载是否太慢、按钮是否难找、报错话术是否看得懂、移动端是否可用。
4. **优化建议（能不能更好）**：你觉得可以更方便的地方，哪怕功能是好的。欢迎给思路。

> 小提示：**免费服务器闲置约 15 分钟会休眠**，第一次打开可能要等 1 分钟左右冷启动，属正常，不用报 bug；但如果**常态化**很慢要记下来。

---

## 二、测试前的准备

1. **注册 3~4 个账号**（注册必填邮箱），用来模拟不同角色，例如：
   - A：数据集**总管理员**（自己建一个数据集）
   - B：数据集**普通成员**（申请加入 A 的数据集）
   - C：**非成员/游客视角**（不加入，测"看得到什么、下不了什么"）
   - D：**课题组管理员**（建一个课题组）
2. 建议准备一份**小的真实 `.dta` 文件**（几百行即可）用于测上传、变量抽取、在线分析；再准备一个 CSV/Excel 当"对照表"。
3. 测试时**开着浏览器控制台**（F12），遇到报错顺手截一下 Network/Console 的红色报错，对定位帮助很大。（有问题时再看）
4. 建议至少在**一台电脑**上走一遍主流程。（手机上大致过一遍就行，现在初步发现手机上的页面显示并不美观）

## 三、分模块测试清单

每个模块给出「功能要点（必测）」和「体验要点（重点体会）」。

### 1. 账号与登录
- 功能要点：注册（用户名/邮箱是否**唯一**，重复时提示是否清楚）；登录/退出；**找回密码**（收没收到邮件、链接能不能重置成功）；长时间后令牌过期是否有合理表现。
- 体验要点：错误提示是否说人话（如"用户名已存在""邮箱已被注册"）；必填项校验是否友好。

### 2. 数据集首页与搜索
- 功能要点：进来能否看到"我参与的 + 发现"两块；**按名称或 ID 搜索**是否命中；"近期更新"条点进去是否跳到对应数据集。
- 体验要点：卡片信息够不够看（名称/ID/归属/更新）；空状态（没有数据集时）提示是否清楚。

### 3. 课题组
- 功能要点：建组（名称是否**唯一**）；申请加入 + 管理员审批；组内动态；**在组内建数据集**（只能建到自己已加入的组）；编辑组/成员管理。
- 体验要点：课题组和数据集的从属关系是否好理解；**负责人（总管理员）+ 邮箱**展示是否正确（应＝当前总管理员及其注册邮箱，转让后自动更新）。

### 4. 数据集详情 · 概览 / 更新记录
- 功能要点：概览的当前版本、公约（管理员能否内联编辑）、**负责人（当前总管理员）+ 邮箱**、成员（默认显示 3 个 + 检索）；更新记录时间线的**筛选**（全部/版本/勘误/代码）。
- **负责人及邮箱自动化（新）**：概览显示的负责人应＝当前总管理员，邮箱＝其注册邮箱（不再手填）；**转让总管理员后，负责人和邮箱应自动变成新人的**。创建/编辑数据集时不应再要求填"联系方式"。（注意：种子演示账号无注册邮箱，做负责人时邮箱可能为空，属正常。）
- 体验要点：页签是否好找；成员点名字能否跳主页。

### 5. 版本库（重点模块）
- 功能要点：
  - 发版：版本号**不可覆盖**（重复号是否报错）；数据分类**原始/脱敏/样例**；上传 `.dta`、Codebook、**对照表**。
  - 下载：`.dta` / Codebook / **对照表** 三种下载是否正常；**样例数据**是否所有登录用户可下；历史版本/当前版本权限是否符合预期。
  - 数据分类标签、"当前版"标记、筛选（全部/原始/脱敏/样例）是否正确。
  - **删除版本（新，仅管理员）**：每条版本后有红色"删除"按钮，二次确认后删除。验证：删除后该版本及其发布记录、下载入口都消失；若删的是**当前推荐版**，系统应自动把剩下最新的原始/脱敏版顶上来当当前版；引用该版本的勘误记录本身应仍在（只是不再指向该版本）。普通成员**看不到**删除按钮。文件已丢失的旧版本也应能正常删掉、不报错。
- 体验要点：**一个重要现实问题**——免费服务器无持久磁盘，**重启后上传的文件会丢**。如果你昨天传的数据今天下载/分析提示"文件可能已丢失"，这是已知限制（不是你操作错），但请记录发生频率，这是我们要不要买云存储的依据。

### 6. 数据处理设置与变量抽取（重点，新功能）
- 功能要点：上传**原始** `.dta` 后，"数据处理设置"里的**变量清单是否自动出现**（应等于你 .dta 的列名，带上 Stata 变量标签）；点"从最新原始数据刷新变量"是否同步；设唯一 ID 变量、脱敏动作能否保存。
- 体验要点：抽取结果提示（新增/保留/停用多少个）是否清楚；换一版数据后变量是否正确更新。

### 7. 数据看板与在线分析沙箱（重点，新功能）
- 功能要点：
  - 进"数据看板"是否显示"已连接原始数据 + 变量清单"；未发布原始数据时是否提示去发布。
  - **在线分析**：用 `result = df.columns.tolist()` 看变量、`result = df['某变量'].describe()` 看分布，能否跑出**真实结果**；AI 生成代码能否运行（AI 需后台配了 key）。
  - 安全：试着写 `open(...)`、`import os` 这类应被拒绝；确认分析**绝不改动原始数据**。
- 体验要点：结果展示是否好读；行数被截断（大数据只读前 5 万行）时是否有提示；报错话术是否能看懂。

### 8. 勘误（三类，注意区分）
- 功能要点：
  - **原始数据勘误**：单条 + 批量（下模板→填→导入）；逐条 AI 打分/终审；**采纳后一键应用改数生成新版本**；"本人提交的勘误需另一名管理员审"这条规则是否生效。
  - **Codebook / 对照表勘误**：在版本库文件旁点"报错"提交，管理员在版本库上方**采纳/驳回**（这类不记历史、逻辑简单）。
  - **处理代码勘误**：代码评论里的勘误标记 / 代码 Bug 流程。
- 体验要点：三类勘误入口是否容易分清；采纳/驳回后状态是否即时更新、是否进消息中心。

### 9. 处理代码库
- 功能要点：上传代码、版本迭代（发布必须写 changelog）、下载、评论、作者授权他人"可改/可重发布"。
- 体验要点：版本/权限/评论的组织是否清楚。

### 10. 文献地图
- 功能要点：三种上传（单条手动 / 引文自动匹配 / 批量）；**AI 真实性核验**（可疑时"确认真实"能否强制上传）；**同集查重**（同一数据集内**标题+作者+年份+刊物四项全一致**才算重复，仅标题相同应允许上传）；AI 综述、词云、导出。
- 体验要点：词云是否好看好用；AI 综述是否有价值；上传浮层三选一是否好懂。

### 11. 研究讨论区 / 其他协作 / 工作台
- 功能要点：发帖（可关联数据集、评论的评论）；**可见范围四选一**（全平台/课题组/数据集/仅自己，可多选）是否真的按范围控制可见性；其他协作里的 Skill 上传/下载/评论；工作台时间轴条目 + 邀请成员。
- 体验要点：可见范围设置是否好理解；不同账号交叉验证"该看到/不该看到"。

### 12. 成员与权限（重点，最容易出问题）
- 功能要点（强烈建议用 A/B/C 三个账号交叉测）：
  - 加入审批；**加入 ≠ 自动能下载/分析**（默认无权限，需单独授权）；
  - 单独授权（下载/在线分析等 8 项）及**到期规则**（永久/指定日期/指定版本/指定项目）；撤销授权；
  - **增/删管理员、转让总管理员**：只有总管理员能操作；总管理员不能直接被移除、要退位须先转让；至少保留一名管理员。
- 体验要点：授权项名称是否好懂；权限变化后对应按钮是否即时显隐。

### 13. 管理后台
- 功能要点：只按"我管理的组/集"切换；成员贡献度/近 30 天活跃度；内联审批；平台管理员的新增/交接/移除（仅平台总管理员可交接与移除，且不能移除自己）。
- 体验要点：范围切换是否清楚；普通用户是否**看不到**不该看的系统标签。

### 14. 消息中心与邮件提醒
- 功能要点：右下角铃铛红点；五类消息（待我处理/权限与身份/互动·评论/协作·工作台/发布·数据与代码）是否分类正确、点击是否跳对页面；60 秒自动刷新；每日摘要邮件（若开了 SMTP）。
- 体验要点：待办是否真的"只给该处理的人"看；红点计数是否准。

### 15. 通用体验（贯穿所有页面）
- 移动端适配、加载速度、双语文案是否有遗漏、按钮/图标是否统一、报错是否友好、刷新后数据是否还在、浏览器前进/后退是否正常。

## 四、不同角色各自的关注重点

| 角色 | 最该盯的 |
|---|---|
| **数据集总管理员** | 发版与版本不可覆盖、删除版本、变量抽取、勘误采纳与应用改数、授权与审批、增删/转让管理员（转让后负责人/邮箱自动更新）、数据集设置与关闭 |
| **数据集普通成员** | 加入后"能做什么/不能做什么"、申请权限与下载申请、提交勘误、在线分析授权、可见范围 |
| **课题组管理员** | 建组查重、加入审批、组内建数据集、成员管理、组内动态 |
| **平台管理员** | 管理后台范围切换、平台管理员的新增/交接/移除边界、系统标签可见性、审计留痕 |
| **非成员/游客** | "看得到 vs 下不了"边界：能否看变量/公开 codebook、样例数据是否可下、原始数据是否被正确拦截 |

> 建议每个模块都用**"该有权限的人"和"不该有权限的人"各点一遍**，这类越权/漏权问题最值钱。

## 五、反馈格式（请统一按此填，便于我们复现修复）

每个问题一条，建议用表格或按下面字段列清楚：

```
【标题】一句话概括，如：上传原始数据后变量清单没有自动更新
【模块】版本库 / 数据处理设置（对应本文第三节编号，如 6）
【维度】功能实现性 / 正确性 / 舒适度 / 优化建议（四选一）
【严重级别】阻断 / 严重 / 一般 / 轻微 / 建议（定义见下）
【账号与角色】用哪个账号、什么身份（如：B，数据集普通成员）
【环境】设备+浏览器（如：iPhone Safari / Win Chrome）、大概时间
【复现步骤】1... 2... 3...（越具体越好，能让人照着重演）
【预期结果】我以为会怎样
【实际结果】实际怎样（附截图/录屏，控制台报错更佳）
【优化思路】（可选）你觉得怎么改更好
```

**严重级别定义：**
- **阻断**：核心流程走不下去（登录不了、发版失败、数据丢失、权限完全失效）。
- **严重**：重要功能错误或数据/权限不一致（算错数、越权能下载、状态不更新）。
- **一般**：功能可用但有明显缺陷（提示缺失、偶发报错、样式错乱）。
- **轻微**：不影响使用的小瑕疵（文案、对齐、错别字）。
- **建议**：纯优化想法。

> 一份好反馈的黄金标准：**别人照着你的"复现步骤"能重现出同样的问题。** 无法复现的问题最难修。

## 六、优化建议怎么提（给你一点方向）

提优化时，除了"这里不好用"，最好带上**为什么**和**建议怎么改**，例如：
- "三类勘误入口分散，测试时分不清 → 建议在一个地方用标签统一入口，或加一句说明各自用途。"
- "在线分析报错只显示英文异常 → 建议把常见错误翻译成中文提示，并给个示例代码。"
- "授权项有 8 个，名字偏技术 → 建议每项加一句大白话说明，或按'下载类/分析类/维护类'分组。"

这类"问题 + 场景 + 思路"的反馈，我们能直接落地。

## 七、已知限制（不用当 bug 报，但可评估影响）

- **冷启动慢**：闲置后首次访问约等 1 分钟。
- **上传文件重启会丢**：免费服务器无持久磁盘。若这点影响大，我们会评估买云存储/对象存储。
- **AI 功能依赖后台密钥**：未配置时 AI 综述/核验/分析代码生成会显示"AI 未启用"，但不应阻断上传等主流程——如果 AI 没配却**卡住了流程**，这要报。
- 邮件提示功能尚未连接真实邮件服务商提供。
