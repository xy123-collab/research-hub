# 科研数据平台对比与差异化定位

> 目的：盘点理工科与通用科研数据协作平台，对比其定位与功能，明确本平台（面向人文社科课题组的数据协作平台）可借鉴什么、必须差异化什么，以及人文社科为什么需要一个自己的平台。
>
> 调研时间：2026-07。对照对象为本项目《COD数据协作平台-项目开发文档-v1》所定义的产品。

---

## 一、平台盘点：六类玩家

科研数据平台不是一个统一市场，而是六类定位很不一样的产品。逐类看清楚，才能知道我们在跟谁比。

### 1. 通用数据仓储（存档发布型）

| 平台 | 背景 | 定位 |
|---|---|---|
| **Zenodo** | CERN / 欧盟 OpenAIRE | 免费通用仓储，每次发布自动分配 DOI，支持版本化记录，数据+软件都收；单次上传上限约 50GB |
| **Figshare** | Digital Science（商业） | 通用仓储 + 机构版订阅，单次可到 10TB，支持公开/受限访问，人工策展需额外付费 |
| **Dryad** | 非营利，期刊联盟 | 与期刊投稿流程绑定，收费但含人工策展（curation），面向"论文配套数据包" |
| **Dataverse（哈佛）** | 哈佛 IQSS，开源 | 机构可自建的仓储系统；数据集版本号 + DOI 不变、文件级受限访问/禁运（embargo）、下载前可要求申请者说明用途。北大开放研究数据平台即基于 Dataverse 搭建 |
| **ScienceDB 科学数据银行** | 中科院网络信息中心 | 国内通用仓储，面向论文关联数据的长期保存与出版，Springer Nature 推荐仓储之一 |

**共同点**：核心动作是"发布一个冻结的数据集，给它一个 DOI，让别人能引用"。数据是**成品**，平台是**图书馆**。
**共同短板**：发布之后数据基本静止。没有勘误流程、没有围绕数据的持续协作、没有细粒度的贡献计量——署名单位是"整个数据集"。

### 2. 开放科学工作流平台

**OSF（Open Science Framework）**，Center for Open Science 出品，是最接近"科研协作"定位的通用平台：项目制组织、协作者权限管理、自动文件版本控制、操作日志、wiki 与评论、预注册（registration）、DOI，并可挂接 GitHub/Dropbox/Google Drive 等外部存储。

OSF 是本平台最值得研究的单一对象：它证明了"项目 + 权限 + 版本 + 日志"这个骨架在科研场景成立。但它对**数据本身**是不可知的——文件就是文件，OSF 不理解表格里的变量、不做数据质量核验、不做贡献定量计量，也没有"数据集"作为一等公民的治理结构（发起人/维护者/勘误/发版）。

### 3. 理工科领域库（学科垂直型）

GenBank（基因序列）、PDB（蛋白质结构）、OpenNeuro（脑影像）、Materials Project（材料计算）、PANGAEA（地球环境科学）等。特点：

- **强制的领域数据标准**：提交必须符合统一 schema（如 BIDS、mmCIF），机器可校验；
- **提交即质检**：入库前有自动格式校验 + 人工/半自动审核；
- **社区围绕数据长出来**：期刊要求投稿必须先存入这些库，形成"不存不能发"的闭环。

这类库能做到强标准，是因为理工科数据天然同构（一条序列就是一条序列）。人文社科做不到全学科统一 schema，但"提交即校验、标准即门槛"的思路可以在**单个数据集内部**复刻——这正是我们 codebook + 数据核验专栏的逻辑。

### 4. 实验室电子记录本（ELN/LIMS）

**Benchling**（分子生物学，样本-实验-结果全链路溯源、协议模板、审计追踪）、**LabArchives**（实验记录模板化、条目不可篡改、审计友好）。定位是"实验过程的过程性记录"，卖点是**溯源与合规**：每条记录谁写的、何时写的、改过什么，全部留痕且不可抵赖。

对我们的启发不在功能而在理念：**溯源本身就是产品核心卖点**，可以支撑商业化定价。我们的审计中间件 + 版本不可覆盖 + 勘误留痕，本质是把 ELN 的合规逻辑搬到了数据资产上。

### 5. 数据科学社区（社区激励型）

- **Kaggle**：数据集 + 竞赛 + notebook，靠 Novice→Grandmaster 等级体系游戏化激励，社区通过 upvote 和讨论对数据集做质量分层；
- **Hugging Face Datasets**：每个数据集是一个 git 仓库，可开 PR、提 issue、按 revision 固定版本复现；社区贡献直接改进数据资产本身，配 dataset card（相当于标准化 codebook）和在线 data viewer。

这两家解决了仓储型平台不管的问题：**让别人愿意持续贡献**。Hugging Face 的"数据集 = 可协作的仓库 + PR 工作流"几乎就是我们勘误管线的工程师版本；Kaggle 证明了积分/等级对科研人群同样有效。但它们的贡献计量都停留在"社交货币"（点赞、下载量、等级徽章），**不折算成学术信用**，也没有敏感数据治理。

### 6. 代码化数据版本工具

Git + **DVC** / git-annex / **DataLad**：把数据当代码管理，diff、branch、回滚。技术上最彻底，但使用门槛是命令行 + Git 心智模型，在非计算背景的人文社科团队内基本推不动。它们的价值是提醒我们：**版本语义（什么算 major/minor/hotfix）比版本工具更重要**——我们用 `v1.0.1 / v1.1.0 / v2.0.0 / -hotfix` 的显式语义化版本，就是把 DVC 的能力翻译成社科研究者能理解的形态。

### 7. 参照系：社科自己已有什么

- **ICPSR**（密歇根大学）：全球最大社科数据档案（25万+数据文件），专业策展团队做披露风险审查、元数据、质量审核；受限数据走**虚拟数据隔离区（VDE）**或安娜堡现场物理隔离区。它是社科数据治理的天花板，但定位是**中心化档案馆**：课题组把数据"交出去"，而不是"在里面持续协作"。
- **北大开放研究数据平台**：基于 Dataverse，机构级发布仓储，同样是存档发布型。
- **CnOpenData / CSMAR / CNRDS**：商业数据供应商，卖标准化成品数据，用户只是买家，无协作无归属。

**结论**：社科世界里，"档案馆"（ICPSR/北大平台）和"商店"（CnOpenData）都有了，缺的正是**中间层——课题组自建数据的协作生产车间**。

## 二、横向对比表

| 维度 | Zenodo/Figshare/Dryad | Dataverse | OSF | 领域库(GenBank等) | Benchling/ELN | Kaggle/HF | ICPSR | **本平台** |
|---|---|---|---|---|---|---|---|---|
| 核心对象 | 冻结的数据集成品 | 数据集成品 | 项目文件 | 标准化记录 | 实验过程 | 数据集+模型 | 策展档案 | **活的、持续修订的数据资产** |
| 版本控制 | 有（记录级） | 有（数据集级） | 有（文件级自动） | 有 | 条目不可改 | git revision | 有 | **语义化版本+不可覆盖** |
| 细粒度权限 | 弱（公开/受限） | 中（文件级+申请制） | 中（协作者角色） | 弱 | 强 | 弱 | 强（隔离区） | **三级权限+隐私隔离（总管理员看不到数据内容）** |

## 三、可借鉴的（逐条对应到功能）

1. **OSF 的项目骨架**：项目-协作者-权限-日志-版本的组织方式直接可抄；它的"挂接外部存储"思路对应我们的 `storage` 双实现（本地/COS/校内）。
2. **Dataverse 的访问分层**：文件级 restricted + embargo + "下载前要求说明用途"，对应我们的下载分级与数据集公约门禁；其"DOI 不变、版本号递增"的引用模型值得在二期引入（给数据集发 DOI，让引用可计量）。
3. **Hugging Face 的 PR 式协作**：dataset card = 我们的 codebook；PR/issue = 我们的勘误提交；revision 固定 = 我们的版本引用。差别在于我们把"merge 权"程序化成了评分制审核，比 maintainer 一个人拍板更适合学术场景。
4. **Kaggle 的激励设计**：等级/徽章的游戏化分层可作为贡献分之上的展示层（个人简历页），但要避免纯社交货币化——我们的分数必须锚定学术信用（作者排序、致谢、数据论文署名依据）。
5. **领域库的"提交即校验"**：入库前机器校验格式/编码/缺失值/ID 唯一性，把质检前置，对应我们的数据核验专栏与定时质量巡检。
6. **ELN 的审计不可抵赖**：所有写操作留痕、条目不可篡改，是敏感数据场景下说服机构（学校/伦理委员会）的关键卖点，对应我们的审计中间件。
7. **ICPSR 的隔离区思想**：不能下载、只能在受控环境内分析——对应我们的只读沙箱看板（非成员在沙箱里跑描述统计但拿不走微观数据），这是用轻量方式复刻 VDE。

## 四、必须差异化的（我们的空白点即定位）

上表最右列已给出结论，浓缩为四条别人都没做的事：

1. **数据是"活的"**。所有仓储平台的对象是冻结成品；我们的对象是长期演化中的自建数据库（如 COD 官员数据库，逐年扩展、持续勘误）。"发布"在别人那里是终点，在我们这里是里程碑。
2. **贡献可归属、可定量、可兑现**。从 Zenodo 到 ICPSR，没有任何平台回答"清洗了三个月数据的博士生拿到什么"。我们的评分制勘误 + 加权贡献分 + 个人简历页，把数据劳动变成可写进 CV 的学术信用。这是冷启动的核心抓手：没有归属激励，课题组根本不愿把数据放上来。
3. **隐私边界内建于权限结构**。别人的管理员是最高权限；我们的总管理员**看不到**课题组数据内容——这个反直觉设计恰恰是敏感数据（官员、微观调查）场景下课题组敢入驻的前提。加上"校内部署放真数据、商业云只放合成数据"的部署边界，治理粒度超过所有通用平台，仅次于 ICPSR，但比 ICPSR 轻得多、且不要求交出数据主权。
4. **协作车间 + 学术社区一体**。仓储没有社区，Kaggle 有社区没有学术性。我们把数据集治理（勘误/发版）和学术交流（研究广场、项目工作台、Overleaf/GitHub 链接、Skill 共享）放在同一处，让"围绕数据的讨论"直接沉淀为"对数据的修订"。

## 五、人文社科为什么需要自己的平台

理工科的数据基础设施繁荣，建立在三个人文社科不具备的前提上：

**前提一：数据同构，所以能标准化。** 一条基因序列在任何实验室含义相同，因此 GenBank 能强制统一 schema、机器自动质检。社科数据高度异构：同是"官员任期"，不同课题组的口径（任职起算日、代理期是否计入）都不同。**社科数据的质量不由格式决定，而由编码决策决定**——所以我们需要的不是全局 schema，而是数据集级的 codebook + 人机结合的勘误审议（一条勘误值不值得采纳，往往是学术判断，这正是评分制审核存在的理由，也是纯技术平台做不了的）。

**前提二：数据来自仪器，所以溯源是自动的。** 测序仪产出自带元数据；社科数据来自人工搜集（年鉴、档案、爬虫、手工核对），每个数值背后是一次人的判断。溯源必须靠制度设计（版本不可覆盖、勘误留痕、来源标注）而不是仪器日志。**溯源在理工科是免费的副产品，在社科是需要平台专门生产的稀缺品**——这恰恰意味着做出来价值更高。

**前提三：署名文化覆盖数据劳动。** 理工科有数据论文、有 consortium 署名惯例，实验室层级会把数据工作折进作者名单；社科（尤其国内）的数据搜集清洗劳动长期隐形，学生做完就走，数据死在毕业硬盘里。这是社科自建数据库难以持续的第一原因，也是"贡献可归属兑现"作为价值核心的依据。

此外三条社科特有约束，决定了不能直接搬用现有平台：

- **敏感性是常态而非例外**：官员数据、入户调查、访谈文本，几乎所有有价值的社科自建数据都碰隐私或政治敏感性。Zenodo/HF 的默认公开逻辑完全不适用；需要的是默认封闭、分级开放（非成员只见 codebook 和聚合看板）。
- **数据的价值在关联而非体量**：社科数据不大（COD 全库远小于一次测序产出），但价值密度高、且靠跨数据集关联（官员×企业×政策）增值。平台重点不是存储与算力，而是变量级元数据、ID 体系与跨集联查（二期）。
- **用户不是工程师**：DVC/HF 的 git 心智模型推不动；界面必须把版本、勘误、审核翻译成学术工作流语言（"提交勘误"而不是"开 PR"）。

## 六、一句话定位

现有平台是**图书馆**（Zenodo/Dataverse/ICPSR，存成品）、**记录本**（Benchling，记过程）或**广场**（Kaggle/HF，聚社区）；本平台是**数据的生产车间与产权登记处**——面向人文社科课题组的自建数据，让它可信（核验+版本+审计）、可控（三级权限+隐私隔离+部署边界）、可归属（评分制勘误+定量贡献+学术信用兑现），并让围绕数据的学术交流直接反哺数据质量。
